VerseLog, from zero to a real, packaged app on two platforms

A free, open-source logistics assistant for Star Citizen. Built solo-dev-and-agent through the BMad method — brainstorm → spec → architecture → epics → story-by-story implementation, each behind its own branch, PR and code review. This page retraces the whole build in order.

Why this exists Spec — Why

A solo, self-taught Star Citizen player was losing more time hand-calculating cargo runs — which contracts to take, how to route them, how to load the ship — than actually playing. VerseLog reads the contract board, validates what it reads, and hands back a route and cargo plan maximizing aUEC earned per minute.

Two constraints shaped every decision that followed: the game is still in alpha with no official data feed and a UI that can change under any patch, and the intended audience runs on modest hardware that can't absorb a heavy companion app on top of an already CPU-bound game.

No paid cloud API, ever. No game-state or memory reading — passive screen capture only. Never auto-accepts or auto-declines anything in-game. Open-source and MIT-licensed from the first commit.

Architecture Ports & Adapters

One domain core that never imports a concrete adapter. Every capture method, data source, trigger, and UI is a swappable implementation of a core-defined interface.

capture/OCRProvider · VisionProvider
datasource/CommunityAPIProvider · LocationDataProvider
trigger/ManualTrigger · VoiceTrigger
ui/UIPort — no adapter yet
core/ — TrustLayer · RouteCostCalculator · LoadingPlanCalculator · CombinedRoutePlanner · ReputationStore · LegalityChecker · SettingsStore
DecisionRule
AD-1Core never imports a concrete adapter. Adapters depend on core-defined ports, never the reverse.
AD-2Python 3.12+ floor — OCR/vision library ecosystems lag the newest release.
AD-3Every provider's raw extraction passes through one trust-layer service before anything downstream sees it.
AD-4Bulk-imported community reference data lives in one local SQLite file — no scattered copies.
AD-5Single long-running desktop process; models load lazily at scan time, unload after.
AD-6Capture runs on 1..N workers; every result is queued into the trust layer serially.
AD-7One local settings store (time budget, reputation, fuel overrides, benchmark results) — adapters never write it directly.
AD-8The benchmark outputs model tier and safe worker count together, in one pass.

Epic 1 — Reliable Contract Scanning 7 stories · PR #1–7

Trigger a scan, get back validated, trustworthy contract data — one contract or a few dozen — within a time budget.

1.1Project Scaffolddone
The Ports & Adapters tree — core/, adapters/{capture,datasource,trigger,ui}/ — with empty port interfaces and a passing hello-world test. Everything else builds on this shape.
1.2Manual Capture via Classic OCRdone
OCRProvider screenshots the screen, runs Tesseract, and a hand-written regex parser (parse_contract_text) extracts departure / arrival / SCU / reward / remaining time. Scoped to Hauling-type contracts — the one real example on file.
1.3Trust Layer — Validation & Quarantinedone
The single service every CaptureResult funnels through. A suspect field (unknown-looking station name, non-positive SCU or reward) quarantines the result and keeps the source image instead of showing it as trustworthy.
1.4Voice Trigger (VoiceAttack, Windows)done
Same trigger/capture composition pattern as manual triggering — a trigger adapter is built with a capture-port dependency and delegates to it. Windows-only; Linux keeps the manual path with no error for the missing voice one.
1.5Local Vision Provider (Ollama)done
VisionProvider asks a local Ollama vision model for structured JSON matching Contract's shape — same port, same return type as OCR, swappable behind CapturePort with nothing downstream needing to know which one ran.
Revisited after real testing — see field logThe assumed model didn't exist and its stand-ins hallucinated; fixed once Ollama was actually installed and run.
1.6Hardware Benchmarkdone
Benchmarks the host while the game is running — an idle-PC reading would flatter a machine that can't keep up mid-session. Picks a model tier and a safe worker count together, never leaving no provider chosen.
1.7Batch Scanning (Parallel Workers)done
A pool of 1..N workers captures in parallel, but every result is handed to the trust layer serially via a calling-thread as_completed() loop — no worker ever writes quarantine state directly.

Field verification Real screenshots, real Ollama, real bugs

Unit tests use fixtures. These entries are what happened when the pipeline met two real Star Citizen screenshots and an actually-installed Ollama — the moment this project's whole "don't invent unverified data" discipline paid for itself.

OCR · reward field

The ¤ currency glyph doesn't survive Tesseract

Against the real Hauling-contract screenshot, Tesseract dropped the ¤ symbol entirely on one image and misread it as & on another — the regex anchored on the literal glyph never matched.

Reward extraction failed — found via real image

Fixed by anchoring on the word "Reward" instead of the symbol: Reward\D*?([\d,]+).

OCR · location field

A wrapped location name carries its line break into the data

"Greycat Stanton IV Production Complex-A" wraps across two lines in the real UI — OCR handed back a raw \n inside the arrival string, which would silently break exact-name lookups against the location store built two epics later.

Arrival field had an embedded newline

Fixed with a _clean_location_name() whitespace-collapse helper, applied to both departure and arrival.

Vision · model selection

Four Ollama vision models, tested against the same two screenshots

The assumed default (phi3-vision) turned out not to be a real Ollama tag at all. Installed Ollama for real and ran all four candidates through both a free-form "describe this image" prompt and the actual structured-JSON capture path.

llava-phi3
Hallucinated completely — described a car dashboard console for a contract screenshot.
moondream
Hallucinated too — repetitive unrelated text, near-placeholder JSON output.
llama3.2-vision
Failed outright: unknown model architecture: 'mllama' on the installed Ollama build.
qwen2.5vl:3b
Read the real content correctly — correct reward, correct locations. Adopted as the new default.
Vision · SCU & context window

Two more bugs surfaced by the same real run

The winning model returned the current SCU amount instead of the capacity ("0" instead of "6" from "0/6 SCU") despite the schema already saying "the total" — fixed with a worked example in the prompt instead of an abstract description. A busier contract screenshot then exceeded Ollama's 4096-token default context window outright.

Both fixed — prompt reworded, num_ctx raised to 8192
Combined route planner · tie-break

An independent verification agent traced a false-rejection bug by hand

When a pickup and a delivery tied on distance, the greedy planner always preferred picking up more cargo — which could stack missions unnecessarily and reject an equal-cost, capacity-feasible route as "exceeds capacity."

Confirmed with a concrete input: two missions, ship capacity exactly matching one mission's SCU

Fixed by listing the unload candidate first, so ties favor delivering before loading more.

Epic 2 — Route & Cargo Optimization 5 stories · PR #8–14

Turn Epic 1's validated contracts into a real route and loading plan, adapted to the ship actually flying it.

2.1Reference Data Import (Local SQLite)done
CommunityAPIProvider paginates api.star-citizen.wiki's vehicle list into ShipReferenceStore — cargo capacity, quantum fuel/range/speed/spool-time, upserted on refresh.
2.2Point-to-Point Route Cost Calculationdone
RouteCostCalculator computes real distance/time/fuel from location coordinates and a ship's quantum stats — 3D Euclidean distance, spool time + distance/speed, fuel-per-meter from the ship's own full-tank range.
Discovery mid-storyapi.star-citizen.wiki has no coordinates at all. Found the real source instead: scunpacked-data's starmap_positions.json, verified to include this project's own captured contract's exact locations. Code review then caught a real ZeroDivisionError for ships with no quantum drive (ground vehicles like the Nox).
2.3Loading Plan — Single Missiondone
LoadingPlanCalculator composes RouteCostCalculator rather than re-validating locations/ship itself — calling it without a working route is structurally impossible. Adds the first SCU-vs-cargo-capacity check in the codebase.
Code review findingFetched the same ship twice from SQLite. Fixed by adding a ship field to RouteCost itself, so the already-fetched ship comes back for free.
2.4Combined Route & Loading — Multiple Missionsdone
Greedy nearest-neighbor construction whose candidate set at every step is only "an unvisited pickup" or "the current top-of-stack's delivery" — makes the LIFO loading order structurally impossible to violate, derived from the same pass that builds the route. An exact TSP/ILP solver was deliberately rejected as NP-hard overkill for a realistic handful of simultaneous missions.
Verified numericallySmart ordering costs 30 distance units vs. 40 for the naive "finish mission A, then B" approach that backtracks. See the field log for the tie-break bug this story's own code review turned up.
2.5Fuel Value Override & Resetdone
FuelOverrideStore wraps the settings store with per-ship fuel-per-meter overrides. Wired into RouteCostCalculator as an optional constructor parameter — every existing call site from 2.2–2.4 needed zero changes.

Epic 3 — Legality & Reputation Safety 2 stories · PR #15–16

Warn before a potentially illegal contract, based on reputation kept in sync — without ever touching the game itself.

3.1Reputation Syncdone
ReputationStore holds per-faction standing in the settings store, mirroring FuelOverrideStore's shape exactly. sync() bulk-updates from detected readings and is a no-op on an empty list, so "nothing detected this pass" never wipes what's already known.
Deliberately deferredNo real screenshot of the in-game REP screen exists yet, so no OCR/vision pattern was invented against it. sync() is the seam a future capture adapter plugs into.
3.2Legality Confirmation Before a Risky Contractdone
LegalityChecker flags a contract when a faction's synced standing is at or below an explicit threshold — missing data (unmapped location, unsynced faction) is never treated as risky. UIPort gains confirm_risky_contract(): it only asks and reports the player's choice back, never acts on the contract itself.
Deliberately deferredNo verified location→faction dataset exists, so the mapping is a plain injected dict rather than invented persisted data.

Epic 4 — Application Entrypoint & Results UI 3 stories · PR #20–22

Wire everything Epics 1–3 built into one running app with a real graphical results window — the last gap standing between "tested in isolation" and "something a player can actually launch."

4.1Application Entrypoint Wiringdone
app.run() composes trigger → capture → trust layer → (optional legality check) → UI into one flow, with an early-return pattern across three terminal outcomes: quarantined, declined (legality risk), or the happy path.
Code review findingThe original except ValueError block reset both route_cost and loading_plan to None even when route-cost calculation had already succeeded and only the later cargo-capacity check failed. Fixed by pre-initializing both outside the try and no longer resetting on except — confirmed correct by an independent verification agent.
4.2Results Window (Tkinter UI Adapter)done
TkinterUIProvider finally implements UIPort for real — one label per ScanResult, a large readable font (NFR9). build_results_window() is split out from the blocking show_results()/.mainloop() call specifically so it stays unit-testable.
Code review findingAn unused from tkinter import font import in production code turned out to be a hidden dependency of a test's own side effect. Removed from production; the test now imports tkinter.font explicitly instead.
4.3Risky Contract Confirmation Popupdone
TkinterUIProvider.confirm_risky_contract() via messagebox.askyesno. app.run() wraps the call in a broad except Exception — a broken or closed confirmation dialog is treated as a decline, never as implicit consent to proceed (guards against ConsoleUIProvider's EOFError risk on closed stdin).

Epic 5 — Packaging & Distribution 5 stories done · 2 deferred · PR #24–29

Turn a working source tree into something a player downloads and double-clicks — on both platforms NFR7 actually names.

5.1Windows Packaging (PyInstaller)done
Single-file --onefile build, published as a GitHub Release asset. Preparing this story surfaced a real gap: __main__.py had never overridden app.run()'s ConsoleUIProvider default, so the real CLI had never actually shown the Tkinter window Story 4.2 built. Fixed by defaulting to TkinterUIProvider, with a new --console-ui opt-out.
Real finding, first runWindows Smart App Control blocked the freshly-built, unsigned exe outright — a categorical block with no per-app "Run anyway" override, unlike SmartScreen. Documented transparently in the Release notes rather than hidden; the real fix (code signing) is tracked as Story 5.4, deliberately deferred.
5.2Linux Packaging (via GitHub Actions)done
A local WSL install proved impractically slow on this machine (confirmed via official docs the real download is only ~300MB — the observed speed was roughly two orders of magnitude below the connection's stated throughput). Pivoted to a disposable ubuntu-24.04 GitHub Actions runner instead — no local footprint, no repeat risk.
Real gotcha, found by a failing CI runuv's own managed Python doesn't reliably ship a working tkinter on Linux, regardless of system tk-dev being installed. Fixed by using the system Python instead (python3-tk + uv sync --python-preference only-system) — confirmed by a second, passing CI run, not assumed from documentation alone.
Manually verifiedDownloaded the real v0.1.0-linux Release asset and ran it on a fresh Ubuntu 24.04 Desktop VM — the Tkinter results window rendered correctly on a real display, something CI itself can never prove (no display server on the runner).
5.3Prerequisite Check for Tesseract/Ollamadone
A new PrerequisiteChecker adapter and UIPort.warn_missing_prerequisites() tell the player plainly what's missing, with an install link — purely informational, never installs anything automatically (the same elevation/reliability risk the WSL attempt had just demonstrated firsthand, kept out of VerseLog's own installer by design).
Extended after a real findingManual Linux VM testing hit exactly the gap this story exists to prevent: Ollama installed and reachable, but its vision model not pulled, surfacing as a bare "status code 404" instead of a plain explanation. Extended mid-story to detect this as its own distinct, correctly-worded case rather than lumping it in with "Ollama not installed at all."
5.5Ship Selection via the Results UIdone
A Tkinter ship-selection screen appears when --ship is omitted, populated from ShipReferenceStore's already-imported data. --ship remains fully honored for VoiceAttack/scripted callers — confirmed with an explicit regression test that select_ship() is never invoked when a ship name is given.
Code review findingAn explicitly empty --ship "" (e.g. a VoiceAttack profile whose variable substitution produced blank text) originally bypassed selection silently instead of triggering it. Fixed by treating any falsy ship name — not just None — the same way.
5.6Monitor Selection for Capturedone
A new --monitor flag (0 = all screens combined, the existing default; 1..N = a specific screen), persisted via SettingsStore. Validated against the real connected-monitor count in __main__.py before it can reach take_screenshot(), which is called outside both capture providers' try/except blocks — an unvalidated index would otherwise crash the app uncaught.
Code review findingThe persisted setting was originally skipped whenever a capture_port happened to be injected — a real inconsistency for any future caller combining both, even though unreachable from the shipped CLI today. Fixed so persistence never depends on that.

Deferred, tracked, not forgotten: Story 5.4 (code signing via SignPath Foundation — free for qualifying open-source projects, deliberately queued behind Linux packaging) and Story 5.7 (a RAM-aware benchmark tier, raised after real vision-model inference exhausted a 4 GB test VM's memory and triggered the OS's out-of-memory killer).

Real-world testing log — Epics 4 & 5 Real screenshots, a real VM, real failures

Packaging and distribution is where a project meets machines it wasn't developed on. Every entry below is something that only surfaced by actually running the built artifact somewhere real.

Windows · packaging

The real CLI had never shown the window Story 4.2 built

app.run()'s ui parameter defaulted to ConsoleUIProvider so tests wouldn't need a display — but that same default had silently leaked into the actual packaged executable.

Found while preparing Story 5.1, not by a test — no test exercised the real entrypoint's default

Fixed by defaulting __main__.py to TkinterUIProvider, with --console-ui as an explicit opt-out.

WSL · local Linux attempt

A ~300 MB download that never finished

Attempting a local WSL install for Linux packaging, the Ubuntu package download sat at 33% after roughly 20 minutes on a connection stated at 100 Mbps — confirmed against Microsoft's own docs that the real package is only ~300–340 MB, meaning the observed throughput was roughly two orders of magnitude below expected.

Abandoned by the project's own author rather than pushed through

Pivoted to a disposable GitHub Actions runner instead — no further local Linux install attempted this session.

Windows · distribution

Windows Smart App Control blocked the freshly-built exe

Rebuilding verselog.exe from merged main and running it triggered "we've blocked an app that could put your device at risk" — confirmed via research this is a categorical block for unsigned binaries with no per-app override, unlike the milder SmartScreen warning.

Confirmed by the project's own author on their own machine

Documented plainly in the Release notes and README rather than hidden; the real fix (code signing) tracked as Story 5.4.

Linux · packaging (CI)

Two CI runs before a working Linux build — one failing, one fixed

The first attempt (install tk-dev, let uv provision its own Python) failed with ModuleNotFoundError: No module named 'tkinter' — uv's managed Python simply doesn't reliably ship a working tkinter on Linux, regardless of system packages.

Run 1 — ModuleNotFoundError, tkinter not importable

Fixed by using the system Python instead (python3-tk + uv sync --python-preference only-system) — confirmed by Run 2, which built and verified the executable successfully.

Linux · manual desktop verification

The real vision model wasn't pulled yet on a fresh VM

Running the packaged Linux binary on a fresh Ubuntu 24.04 Desktop VM produced the correct Tkinter window — but the scan quarantined with "model qwen2.5vl... not found, status code 404". Ollama was installed and running; its vision model simply hadn't been pulled yet, a real first-run gap for any new install.

Window rendered correctly — the actual goal of this verification

Directly informed Story 5.3's mid-story extension: detect "Ollama reachable, model missing" as its own distinct, correctly-worded case.

Linux · vision-model inference

Running the vision model for real exhausted a 4 GB test VM

Once the model was pulled and actually invoked for inference (not just launched), the VM's memory was exhausted and the OS's low-memory monitor force-stopped the heaviest process — taking the browser down alongside it. Fine on 4 GB right up until the model is actually used.

A real memory-exhaustion crash, not merely "slow"

Documented as a real hardware consideration for the vision tier; raised Story 5.7 (RAM-aware benchmark tier selection) as a direct result.

Epic 6 — Guided Installer 1 of 3 stories done · PR #30

A separate, guided installer wizard for Windows — the "Next, Next, Finish" experience of a familiar native installer, not a bare executable a player has to already know how to run. Modeled explicitly on Python's own installer and VMware Workstation's. In progress.

6.1Installer Wizard Shell & Benchmark Stepdone
A genuinely separate package, verselog_installer — it may import from verselog.core/verselog.adapters (reusing Benchmark, SettingsStore, the same capture providers), but verselog never imports anything from it. A Tkinter wizard shell (Back/Next/Finish navigation) opens with a Welcome step, then a Benchmark step that runs the exact same benchmark app.py uses and persists the result to the same settings the real app reads.
Code review findingNavigating Back then Next to the Benchmark step a second time called build() fresh (a brand-new label defaulting to "Checking your hardware...") and then on_shown() — which returned immediately since the result was already cached, skipping the label update entirely. The label stayed stuck on "Checking..." forever even though the result was known. Fixed so revisiting the step always refreshes the label from the cached result.

Remaining, not yet built: Story 6.2 (component selection via checkboxes, pre-checked from the benchmark's recommendation, guided installation that launches each prerequisite's own official installer rather than a silent scripted install) and Story 6.3 (finish screen with an optional desktop/Start Menu shortcut). PyInstaller packaging of verselog-installer.exe itself is deliberately deferred until Story 6.3 completes — no Release asset for a functionally incomplete installer.

What's not built yet Stated, not hidden

Deferred deliberately, each with a reason on file rather than guessed at:

–
No code signing yet (Story 5.4). The Windows build is unsigned, so Smart App Control blocks it outright. SignPath Foundation (free for qualifying open-source projects) is the leading candidate; deliberately queued behind Linux packaging.
–
No RAM-aware benchmark tier (Story 5.7). The benchmark picks a capture tier purely by measured time — not by whether the machine actually has enough free memory for the vision model, confirmed to matter by a real out-of-memory crash.
–
No reputation-screen capture adapter. ReputationStore.sync() is ready to receive readings; nothing produces them yet.
–
No verified location→faction dataset. LegalityChecker's territory mapping is an injected dict until a real source is found.
–
Guided installer only partly built (Epic 6, in progress). The wizard shell and benchmark step exist (Story 6.1); component selection, guided installation, and the finish screen (Stories 6.2/6.3) are next.
