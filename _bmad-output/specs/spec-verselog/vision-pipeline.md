# Vision Pipeline — Provider Abstraction & Fallback Chain

Implementation strategy supporting CAP-1 (capture/extraction) and CAP-3 (adaptive performance). Not kernel-shape: this is HOW, cited by SPEC.md rather than embedded in it.

## Provider abstraction

A single internal `Contract` data model (departure, arrival, SCU, reward, ...) sits behind interchangeable provider implementations, so the extraction source can evolve without touching the rest of the app:

| Provider | Role |
|---|---|
| `OCRProvider` | Classic OCR (e.g. Tesseract/EasyOCR) — lightest, least precise, most sensitive to UI transparency. |
| `VisionProvider` | Local vision-language model via Ollama — higher precision (when the model is actually capable), higher resource cost. |
| `CommunityAPIProvider` | Third-party community data source, backed by **two** StarCitizenWiki-maintained sources for two different needs. **Ships/cargo/fuel:** `api.star-citizen.wiki` (github.com/StarCitizenWiki) `/api/vehicles` — `cargo_capacity`, `cargo_grids[].scu`, `cargo_limits`, quantum/fuel fields (`quantum_fuel_capacity`, `quantum_range`, `fuel.usage` by thruster type). Confirmed paginated via standard `links.next` (290 vehicles / 10 pages, verified directly). Its per-vehicle `port_olisar_to_arccorp_time/fuel` fields are a single hardcoded Stanton-only example pair, not a real route matrix. Its `/api/missions` endpoint is a **static mission catalog**, not live per-player contract instances — does not replace CAP-1's OCR/vision capture. **Location coordinates:** `/api/locations` on the same API does **not** expose x/y/z coordinates (confirmed by direct inspection, list and detail views) — real point-to-point route cost instead uses **`scunpacked-data`** (github.com/StarCitizenWiki/scunpacked-data), a raw game-file datamining project by the same org, whose `starmap_positions.json` gives real x/y/z coordinates for 1250 entities across Stanton/Pyro/Nyx plus inter-system jump-point connections with fuel costs — confirmed to include the exact locations from VerseLog's own captured contract example (Port Tressler, Greycat Stanton IV Production Complex-A), last updated 2026-06-02. Both sources' rate limits and redistribution terms are unpublished — implement caching/backoff defensively regardless. The project appears single-maintainer (bus-factor risk): mitigate server-side, not client-side — fork/mirror the API service itself onto an independently-hosted server (free hosting or GitHub-hosted), as a fallback endpoint if the original goes down. VerseLog's client keeps calling "an API" either way and stays lightweight; bundling the full dataset into the app itself was considered and rejected as it would conflict with the modest-PC constraint. |
| `OfficialAPIProvider` | Official CIG data feed, if one is ever released. Currently non-existent — see SPEC.md non-goals. |

Swapping the active provider must not require changes to route/cargo optimization (CAP-4) or any other downstream capability.

## Adaptive model fallback chain and worker pool (CAP-3)

1. Auto-benchmark the most precise available vision model **while Star Citizen is actually running** — automatically on first launch, on detected hardware change, or any time the user manually triggers "Re-run benchmark" from settings. An idle-PC benchmark reads as artificially more powerful and leads to selecting a model that later misses the time budget during real play.
2. If it exceeds the user's time budget (default 30s), fall back one tier. **The originally-assumed three-tier chain (`Phi-3-Vision → Moondream2 → classic OCR`) does not hold up against real screenshots (confirmed 2026-07-09):**
   - `phi3-vision` isn't a real Ollama library tag at all; the real model pairing LLaVA's vision encoder with a Phi-3 backbone is `llava-phi3`.
   - Tested against two real Star Citizen contract screenshots (structured JSON output *and* free-form description prompts): `llava-phi3` and `moondream` both **completely hallucinate** — neither reads the actual screen content (e.g. `llava-phi3` described a car dashboard console for a contract screenshot; `moondream` returned repetitive unrelated text and near-placeholder JSON). Both are ruled out for this use case.
   - `llama3.2-vision` (11B) fails outright on the currently-installed Ollama version (0.31.2): `unknown model architecture: 'mllama'` — an Ollama-side gap, not something this project can fix; also ruled out for now.
   - `qwen2.5vl:3b` is the only model of the four tested that actually reads real screenshot content correctly. It's the current `VisionProvider` default and the only confirmed-viable tier below classic OCR. The fallback chain is provisionally `qwen2.5vl:3b → classic OCR` until a second viable tier is found (candidates not yet tested: `qwen2.5vl:7b`, `minicpm-v`, or re-testing `llama3.2-vision` once Ollama adds `mllama` support back).
   - `qwen2.5vl:3b` also needs `options={"num_ctx": 8192}` (Ollama's 4096 default was measured being exceeded by a busier real contract screenshot) and a prompt that gives a concrete worked example for the SCU field (an abstract "the total, not the current" description in the schema alone was not enough — the model initially returned the current amount instead of the capacity).
   - **Known unresolved gap**: for a contract type outside what the prompt covers (e.g. Mercenary, which has no departure/arrival/SCU concept), `qwen2.5vl:3b` doesn't fail cleanly — it returns a fabricated-but-schema-valid `Contract` instead of signaling "not applicable." This is currently caught only as a side effect of the trust layer's station-name validation (empty departure/arrival fails `looks_like_a_station_name`), not by an explicit "not a hauling contract" signal from the model itself.
3. The benchmark also determines a safe **worker count (N)**: how many contracts can be captured/extracted in parallel without breaking the time budget — needed because a player can have dozens of contracts visible at once (confirmed in practice, not just theoretical). Extraction runs on a pool of 1..N workers; N is re-evaluated every time the benchmark re-runs.
4. Regardless of how many workers extract in parallel, every result is handed to the trust-layer validation/quarantine service **serially** — that service is the only writer to quarantine/confidence state, so parallel capture never races on shared state.
5. The model loads only at scan time (lazy load) and unloads from VRAM immediately after extraction — footprint is a short burst, not a sustained background cost.

Dedicated tests must verify the auto-benchmark switches correctly between all three model tiers, that the worker count adapts sensibly to hardware, and that manual re-benchmarking updates both (see SPEC.md CAP-3 success criterion).
